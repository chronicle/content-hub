# Task 452223618: MP Tool - Case / Alert Widgets Sync (Views Support)

## Background & Goal
Currently, the `mp` tool only supports pushing and pulling alert view templates because they are embedded within playbook JSON files. However, case views are standalone entities in SecOps SOAR. This task bridges the gap by adding dedicated `mp pull view` and `mp push view` commands to allow version controlling Case/Alert views in external Git repositories (under `content/views/`).

---

## Technical Solution

### 1. View Representation & Deconstruction
To make views readable and editable in Git, we deconstruct a built view template JSON payload into a structured directory:
*   `view.yaml`: Contains metadata (identifier, name, creator, playbook_id, type, roles, role_names) and the ordered list of widgets (`widgets_details`).
*   `widgets/<widget_title>.yaml`: Holds the metadata and settings for individual widgets.
*   `widgets/<widget_title>.html`: Contains the raw HTML content if the widget is of type `HTML`.

This is implemented via:
*   `ViewDeconstructor` (`packages/mp/src/mp/build_project/restructure/views/deconstruct.py`): Takes a loaded `Overview` model and deconstructs it to disk.
*   `ViewBuilder` (`packages/mp/src/mp/build_project/restructure/views/build.py`): Reloads a view from its directory, merges the HTML widget contents back, and outputs a single built view template JSON.

### 2. View Types Support
We support standard Case (`SYSTEM_CASE` type 3) and Alert views. *Note: Modern V2 view types (`SYSTEM_CASE_V2`, `SYSTEM_ALERT_V2`, `SYSTEM_DETECTION`) were intentionally excluded from this implementation as they are still in private preview/testing phase on the platform and are not present in the production codebase.*

### 3. Backend REST API Endpoints
We mapped the backend `ViewService` RPC endpoints and added them to `BackendAPI` client (`packages/mp/src/mp/dev_env/api.py`):
*   `list_views()` $\rightarrow$ `GET /api/external/v1/case-overview/GetOverviewTemplateCards` (fetches all installed view template summaries).
*   `download_view(identifier)` $\rightarrow$ `GET /api/external/v1/case-overview/GetFullOverviewTemplateDetails/{identifier}` (downloads the full built view JSON).
*   `upload_view(data)` $\rightarrow$ `POST /api/external/v1/case-overview/SaveOverviewTemplate` (saves or updates a view template on the server).

### 4. CLI Subcommands Registration
We implemented the CLI subcommands using Typer:
*   `mp pull view "View Name or ID"`: Locates the view on the server, downloads it, and deconstructs it to `content/views/<view_identifier>/` (or a custom `--dst` folder).
*   `mp push view "view_identifier"`: Reads `content/views/<view_identifier>/`, builds it into a single JSON, and uploads it to the configured SOAR platform.

These commands are registered in `packages/mp/src/mp/dev_env/__init__.py`. We also registered the commands in the telemetry command mapper (`NAME_MAPPER`) under `"pull_view"` and `"push_view"` to avoid `KeyError` crashes during telemetry reporting.

### 5. Payload Normalization / Denormalization
During testing, we discovered a mismatch between the JSON schema returned by the standalone views API (`GetFullOverviewTemplateDetails`) and the standard schema used in playbooks:
*   **Playbooks** store overviews wrapped under a `"OverviewTemplate"` key, using **PascalCase** keys, and a stringified `"DataDefinitionJson"` for widget configs.
*   **GetFullOverviewTemplateDetails** returns a flat **camelCase** payload with widgets structured in `{ metadata: { ... }, config: { ... } }` blocks.
*   **SaveOverviewTemplate** expects the flat camelCase layout on upload.

To bridge this gap without breaking the core `Overview` domain model (which relies on the standard playbook format), we implemented translation helpers in the CLI layer:
*   `_normalize_downloaded_view` (in `pull.py`): Transforms the flat camelCase server response into the wrapped PascalCase `BuiltOverview` structure prior to model instantiation. This includes explicitly normalizing the `conditionsGroup.conditions` array items from camelCase (e.g. `fieldName`, `value`, `matchType`, `customOperatorName`) to PascalCase (`FieldName`, `Value`, `MatchType`, `CustomOperatorName`) to avoid `KeyError: 'FieldName'` crashes when a view widget has active conditions.
*   `_denormalize_pushed_view` (in `push.py`): Translates the wrapped PascalCase `BuiltOverview` output from the builder back into the flat camelCase structure for uploading, including denormalizing the conditions array items back to camelCase.

### 6. Dynamic ID Resolution on Push (UPDATE vs INSERT)
The C# SOAR backend API determines whether to update an existing view template or create a new one based on the presence of the integer primary key (`id`) in the payload. However, since the database `id` is instance-specific, it is not version-controlled in the Git repository (to keep configurations portable).
To allow `mp push` to perform updates on existing templates:
*   During `mp push view`, the tool first fetches all installed views on the server using `backend_api.list_views()`.
*   It matches the view being pushed by its UUID `identifier` case-insensitively.
*   If a match is found on the server, the tool retrieves the target `id` and injects it into the push payload. This tells the server to perform an `UPDATE` on the existing template instead of creating a duplicate `INSERT` card.

### 7. Code Quality, Lints, and Review Bot Fixes
During code reviews, we implemented several security, type safety, and defensive programming enhancements:
*   **Case-Insensitive UUID Matching**: Standardized view lookup and ID resolution to use case-insensitive UUID comparisons (`identifier.lower()`), preventing duplicate view creation if server and local casing differ.
*   **Frozen Dataclasses Support**: Refactored widget building logic to use `dataclasses.replace` when updating HTML widgets. This prevents `FrozenInstanceError` when `data_definition` is a frozen dataclass, while keeping normal mutation fallbacks for other classes and `dict` configurations.
*   **Empty Title/Identifier Filename Fallback Consistency**: Synchronized filename generation in `ViewBuilder` and `ViewDeconstructor` to use the widget's index `i` (e.g. `widget_{index}`) as a fallback when the widget has an empty title and missing identifier.
*   **Input Formatting Validation**: Added a safety check in `pull_view` to verify that downloaded API payloads (`built_view_data`) are dictionaries before trying to normalize them, preventing runtime `AttributeError` exceptions.
*   **Regex Class Safety**: Corrected `sanitize_widget_filename` character class regular expression to use proper brackets `[...]` to sanitize all platform-unsafe characters without raising path traversing exceptions.
*   **Null-Coercion Protections**: Replaced `.get(key, default)` calls with `.get(key) or default` where null values returned from the server could bypass default parameters. Added explicit `is not None` logic to prevent logical operator coercions (where value `0` for `AND` was incorrectly overridden by `1` for `OR`).

### 8. SaaS API Key Verification Support & Widget Type Coverage
*   **SaaS API Root & Key Verification**: SaaS environments do not support the legacy `GetSourceRepositorySettings` endpoint (which returned a 404 error during login authentication checks), nor do they expose the `GetSystemVersion` settings endpoint. To support remote SaaS instances, the credentials verification check URL in `BackendAPI.login` was updated to call the `/api/external/v1/case-overview/GetOverviewTemplateCards` endpoint, which is guaranteed to be mapped and active for SaaS instances that utilize Views.
*   **Expanded Widget Type Support**: Unified views on SaaS instances introduced several new widget types not present in legacy/local environments. We added support for widget type enum values 22 through 29 (`COMPOSITE_ALERT`, `RULE_OVERVIEW`, `CASE_DETECTIONS`, `UDM_EVENTS`, `UDM_ENTITIES`, `EVENT_HIGHLIGHTS`, `THREAT_HUNT`, `THIRD_PARTY`) to prevent `ValueError: X is not a valid WidgetType` parser crashes when pulling views with predefined SaaS-only components.

## Verification & Testing

### 1. Builder/Deconstructor Roundtrip Unit Tests
We added unit tests in `packages/mp/tests/test_mp/test_build_project/test_restructure/test_views.py` that verify a full deconstruct-and-build roundtrip. It mocks an Overview object containing both regular and HTML widgets, deconstructs it to a temporary directory, asserts the files (YAML and HTML) are correctly structured on disk, builds it back, and validates the reconstructed JSON matches the original built payload.

### 2. CLI Command Integration Tests
We added CLI integration tests in `packages/mp/tests/test_mp/test_dev_env/test_sub_commands/test_views.py` using `typer.testing.CliRunner`. These tests mock the `BackendAPI` client calls:
*   `test_pull_view_cli` mocks `list_views` and `download_view` to verify that `mp pull view` downloads and writes correct deconstructed files to disk.
*   `test_push_view_cli` sets up a mock local view directory, mocks `upload_view`, runs `mp push view`, and asserts that the collected, built view payload sent to the backend API contains all widgets and correct type identifiers.

### 3. Verification Suite
Verified that the package passes typechecking (`uv run ty check`) and formatting checks (`ruff check`). All tests passed successfully.
